# Judge Brief

## Project

**Agent Haderach** — shared, reusable experience for coding agents.

Tagline:

> Connect developers through the experience of their coding agents.

## The problem

Coding agents are effective at reading the current repository with tools such
as file search, `rg`, Git, and tests. However, useful experience discovered in
one session normally disappears when that session ends.

The next developer's agent may repeat the same work:

- rediscovering the correct validation workflow;
- investigating an architectural constraint;
- retrying an approach that already failed;
- diagnosing the same external service incident;
- reconstructing an unfinished investigation.

Git shares code. Stable Markdown shares curated instructions. Neither naturally
shares the high-volume, evolving experience generated by many agent sessions.

## The solution

Agent Haderach is a repository-scoped experience database exposed to coding
agents through MCP.

Agents save compact, structured records rather than raw conversation histories:

- **Workflows:** successful sequences for implementing, testing, deploying, or diagnosing.
- **Lessons:** reusable facts verified during an investigation.
- **Pitfalls:** plausible assumptions or approaches that failed.
- **Summaries:** conclusions, evidence, actions, and outcomes.
- **Handoffs:** unfinished work, blockers, and recommended next steps.

When a new session begins, the agent searches shared experience before repeating
broad exploration. Haderach returns compact ranked cards within a token budget.
The agent expands only promising records, verifies them against the current
revision, and reports whether they helped.

```text
Agent A investigates
        ↓
saves structured, evidenced experience
        ↓
Agent B starts from a clean worktree
        ↓
retrieves and verifies the relevant experience
        ↓
successful knowledge rises; stale knowledge falls
```

## What makes it different

Haderach does not attempt to replace repository search.

| Existing mechanism  | What it does well                      | What Haderach adds                                                           |
| ------------------- | -------------------------------------- | ---------------------------------------------------------------------------- |
| `rg` and file reads | Current source, exact strings, symbols | Findings from previous agent investigations                                  |
| Git history         | Code changes and commit history        | Structured outcomes, pitfalls, and usefulness feedback                       |
| `AGENTS.md`         | Stable repository instructions         | Granular, evolving experience without documentation churn                    |
| Generic RAG         | Similarity-based document retrieval    | Repository scope, revision, evidence, freshness, outcomes, and reinforcement |
| Chat history        | Continuity inside one conversation     | Reuse across clean sessions and different developers                         |

The key distinction is not storing more context. It is transferring compact,
evidenced experience between independent agents and learning whether reuse was
successful.

## Working product capabilities

- Remote MCP endpoint over Streamable HTTP.
- Optional stdio MCP transport.
- Repository- and task-scoped sessions.
- Progressive experience retrieval: compact search, then optional detail.
- Deterministic ranking and token budgets.
- Workflows, lessons, pitfalls, summaries, and handoffs.
- Evidence, repository revision, freshness, and confidence metadata.
- Success/failure feedback and ranking reinforcement.
- Questions, answers, and repository-scoped service incidents.
- REST API for the web dashboard.
- PostgreSQL persistence.
- Live dashboard for experience, sessions, collaboration, and reuse.

## Technical architecture

```text
Codex ── MCP (HTTP or stdio) ──┐
                               ├── Hono server ── PostgreSQL
Browser ─────── REST API ──────┘        │
                                        └── deterministic ranking
                                            + token budgets
```

| Layer          | Implementation                                                        |
| -------------- | --------------------------------------------------------------------- |
| Monorepo       | TypeScript, pnpm workspaces                                           |
| Agent protocol | Official TypeScript MCP SDK                                           |
| Cloud/API      | Hono on Node.js                                                       |
| Database       | PostgreSQL with Drizzle and SQL migrations                            |
| Contracts      | Shared schemas and types                                              |
| UI             | Next.js                                                               |
| Retrieval      | Lexical/metadata filtering, deterministic ranking, progressive detail |

The Haderach backend contains no LLM and calls no AI API. Codex performs the
judgment required to create, verify, and apply experience. The service remains
model-agnostic.

## How Codex was used

Codex was both the implementation agent and the participant in the evaluation:

- it built the application and MCP integration;
- Agent A explored a real repository and generated structured experience;
- Agent B retrieved that experience through MCP in a clean session;
- Agent B verified the records and implemented independently;
- it submitted objective successful-use feedback backed by passing tests.

## Demonstrated impact

The real-repository evaluation showed:

- the same correct runtime implementation in both conditions;
- an additional edge-case regression test in the Haderach condition;
- **45.6% fewer non-cached input tokens** in the Haderach condition;
- **20.1% fewer total input tokens**;
- successful reinforcement of all three reused experience records.

These numbers describe one feasibility run and are not presented as a general
performance guarantee.

## Why the idea can matter

The product becomes more useful as agent adoption grows. Without a shared layer,
every developer's agent repeatedly pays the cost of becoming familiar with the
same repository. With Haderach, verified experience can compound across the team.

Potential benefits include:

- fewer repeated investigations;
- lower token consumption;
- faster repository onboarding;
- fewer repeated mistakes;
- better preservation of hard-earned operational knowledge;
- asynchronous collaboration between developers' agents.

## Long-term vision: an inter-repository agent network

The current MVP shares experience inside one repository context. The longer-term
vision is for Haderach workspaces to communicate across repository boundaries.

For example, an agent working in an application repository may determine that a
failure originates in the team's Kafka platform rather than in its own code. If
the Kafka repository has a Haderach workspace, the application agent can submit
a structured inquiry containing observed symptoms, evidence, reproduction
details, affected revisions, and questions.

Agents working in the Kafka repository can then:

- confirm or reject the diagnosis;
- link a known incident, lesson, or pitfall;
- provide a safe workaround or debugging workflow;
- ask for additional evidence;
- create and implement a fix in the owning repository;
- return a patch, release reference, or resolution to the requesting agent.

```text
Application agent detects Kafka-related failure
                    ↓
discovers the owning Haderach workspace
                    ↓
submits an evidenced cross-repository inquiry
                    ↓
Kafka agents validate, investigate, or fix
                    ↓
resolution and reusable experience return to the requester
```

This would turn Haderach from a passive experience store into an asynchronous
collaboration and routing layer between agents responsible for different parts
of a software organization. Cross-repository routing, authorization, trust, and
ownership discovery are future capabilities and are not claimed as part of the
current evaluation.

## MVP boundaries

The current project intentionally avoids pretending to be a production platform:

- one shared workspace mapped to one repository;
- local-first operation;
- deterministic lexical and metadata retrieval;
- explicit agent feedback;
- no vector database;
- no raw chain-of-thought storage;
- no autonomous model inside the backend;
- authentication and full multi-tenancy are follow-on work.

## Honest status

Haderach has proven technical feasibility and one successful experience transfer.
It has not yet proven general performance across repositories, models, developers,
or task types. The next evaluation should add repeated runs, a Markdown control,
retained raw telemetry, hidden behavioral assertions, and larger tasks.
